## 🧠 Part 1：核心概念速查

### 1.1 max_seq_length = 128 的推導

**問題**：為什麼不用 512（BERT 的最大長度）？

**答案**：這不是規則，而是**成本-效益平衡**。

```python
# GPU 內存計算公式
Memory_per_batch = batch_size × seq_length × hidden_size × precision_bits / 8

# 舉例（DistilBERT）
# hidden_size = 768（DistilBERT 的隱藏層維度）
# precision_bits = 32（float32）

# 若用 max_seq_length=512:
Memory = 8 × 512 × 768 × 32 / 8 = ~1.6GB（僅僅一個 forward pass！）
# 加上梯度、優化器狀態、activation 記憶，至少需要 8GB
# 在 8GB GPU 上，會直接 OOM

# 若用 max_seq_length=128:
Memory = 8 × 128 × 768 × 32 / 8 = ~400MB（舒服訓練！）
```

**IMDB 數據的現實**：
```python
# 實測 IMDB 評論的長度分布
from datasets import load_dataset
dataset = load_dataset("imdb")

token_lengths = [len(tokenizer.encode(text)) for text in dataset['train']['text']]
import numpy as np

print(f"平均長度: {np.mean(token_lengths):.1f}")   # 約 230 tokens
print(f"中位數: {np.median(token_lengths):.1f}")    # 約 180 tokens
print(f"95 百分位: {np.percentile(token_lengths, 95):.1f}")  # 約 320 tokens
print(f"總長度 > 512 的比例: {sum(l > 512 for l in token_lengths) / len(token_lengths) * 100:.1f}%")  # < 2%
```

**結論**：
- IMDB 95% 的評論 < 320 tokens
- 設 128 會截斷 ~3-5% 的長尾評論
- **取捨**：損失 1% 的邊界案例，換來 4 倍內存節省 → 能用 8GB GPU 訓練
- 實務判斷：對於短文本任務（評論、推特、新聞標題），128 是黃金點

### 1.3 Compute Metrics 的三層遞進

**層級 1：最幼稚的版本（不推薦）**
```python
def compute_metrics(eval_pred):
    predictions, labels = eval_pred
    predictions = np.argmax(predictions, axis=1)
    
    accuracy = (predictions == labels).mean()
    return {'accuracy': accuracy}
```
❌ **問題**：
- 只看準確率，忽視了類別不平衡
- 如果訓練數據 60% 正面、40% 負面，模型可以籠統地全猜「正面」達到 60% 準確率
- 對於情感分類，完全沒看到「有多少負面被誤判為正面」

---

**層級 2：有點腦子（我給你的版本）**
```python
from sklearn.metrics import accuracy_score, f1_score

def compute_metrics(eval_pred):
    predictions, labels = eval_pred
    predictions = np.argmax(predictions, axis=1)
    
    return {
        'accuracy': accuracy_score(labels, predictions),
        'f1': f1_score(labels, predictions, average='weighted'),
    }
```
✅ **優勢**：
- F1 = 2 × (Precision × Recall) / (Precision + Recall)
  - Precision：說「正面」時，有多準？
  - Recall：所有真正的「正面」，我能抓到多少？
  - 平衡兩者，不被多數類主導
- `average='weighted'`：按類別比例加權
  - 如果數據 60% 正面、40% 負面
  - F1 = 0.6 × F1_positive + 0.4 × F1_negative

**為什麼 weighted 而不是 macro？**
```python
# 假設：2000 個正面、1000 個負面
# 模型結果：
#   正面識別率：0.90 (F1_pos = 0.90)
#   負面識別率：0.50 (F1_neg = 0.50)

# macro (不考慮數據比例):
#   F1 = (0.90 + 0.50) / 2 = 0.70

# weighted (按比例):
#   F1 = (0.90 × 2/3) + (0.50 × 1/3) = 0.60 + 0.167 = 0.767
#   → 更接近實際性能（因為我們有 2/3 的正面）

# 實務建議：用 weighted
#   - 反映真實業務場景（數據本身就不平衡）
#   - 不會被少數類的完美預測迷惑
```

### 2.1 關鍵函數解析

**第 1 步：數據加載**
```python
def load_data(self, train_file, val_file, test_file) -> tuple:
    """為什麼要分開 train/val/test？"""
    
    # 讀取 CSV
    train_df = pd.read_csv(train_file)  # 8000 行
    val_df = pd.read_csv(val_file)      # 2000 行
    test_df = pd.read_csv(test_file)    # 5000 行
    
    # HuggingFace Dataset 格式
    train_dataset = Dataset.from_pandas(train_df)
    val_dataset = Dataset.from_pandas(val_df)
    test_dataset = Dataset.from_pandas(test_df)
    
    return train_dataset, val_dataset, test_dataset
```

**為什麼不能在測試集上訓練？**
```
訓練集：模型用來學習規律
驗證集：在訓練中檢查模型是否過擬合（early stopping）
測試集：最後評估「真實泛化能力」(不能在訓練中看!)

如果在測試集上訓練或調參：
  → "信息洩露"（data leakage）
  → 測試集評分虛高，真實部署會慘淡
  → 這是深度學習「大忌」！
```

**第 2 步：數據預處理 (preprocess_data)**
```python
def preprocess_data(self, dataset: Dataset) -> Dataset:
    
    # 加載分詞器（跟模型配對！）
    tokenizer = AutoTokenizer.from_pretrained(self.model_name)
    
    def tokenize_function(examples):
        """分詞函數 - 文本 → token ID"""
        
        # examples 是一個 dict，包含 'text' 和 'labels'
        # {'text': ['Review 1', 'Review 2', ...], 
        #  'labels': [0, 1, ...]}
        
        return tokenizer(
            examples['text'],                # 要分詞的文本
            padding='max_length',            # 短句子補 padding 到 128
            truncation=True,                 # 長句子截斷到 128
            max_length=self.max_seq_length,  # 128
            return_tensors='pt'              # 返回 PyTorch tensor
        )
        
        # 返回：
        # {
        #   'input_ids': [101, 2054, 2003, ...],  (128 個 token ID)
        #   'attention_mask': [1, 1, 1, ..., 0, 0]  (真實token=1, padding=0)
        # }
    
    # 把分詞函數應用到整個 dataset
    dataset = dataset.map(
        tokenize_function,
        batched=True,  # 批量處理，快速
        batch_size=1000,  # 一次處理 1000 個
        remove_columns=['text']  # 分詞後不再需要原文本
    )
    
    # 重命名標籤列（HF Trainer 期望的名字）
    dataset = dataset.rename_column('target', 'labels')
    
    # 設置格式讓 PyTorch 能用
    dataset.set_format('torch', columns=['input_ids', 'attention_mask', 'labels'])
    
    return dataset
```

**你應該理解的細節**：
```python
# Padding 和 Truncation 會發生什麼？

# 例子 1：短評論

# 用 max_length=128, padding='max_length'

    'attention_mask': [1, 1, 1, 1, 1, 0, 0, 0, ..., 0]  # 前 5 個是真實，後 123 個是 padding
}

    'input_ids': [101, ..., 102],  # 只保留前 128 個 token，第 129+ 個被丟掉
    'attention_mask': [1, 1, ..., 1]  # 全是 1，沒有 padding
}

# 關鍵：Attention Mask
# - 1: 模型應該看這個 token
# - 0: 這是 padding，模型應該忽視
# → 不然 padding token（全是 0）會污染注意力權重
```

### 2.2 BERT 微調參數懶人包（核心 vs. 調校 vs. 參考）

時間寶貴，不是所有參數都同等重要。這份清單幫你分類哪些「必刻進大腦」，哪些「知道就好」：

#### 🧠 Level 1：核心靈魂（必須深度理解）

**為什麼？** 這三個參數直接決定了你的 3050 顯卡會不會爆炸。

| 參數 | 你的值 | 為什麼重要？ | 記住這個 |
|------|-------|-----------|---------|
| **max_seq_length** | 128 | BERT 計算量 ∝ 長度² → 125→512 顯存 ↑ 16 倍！ | 顯存殺手 |
| **learning_rate** | 2e-5 | 太大→模型學崩；太小→練到天荒地老 | 微調用極小值（2e-5～5e-5） |
| **per_device_train_batch_size** | 8 | 顯存 ↔ 訓練穩定度的平衡點。用 2 的次方（8/16/32） | 記住公式：Memory ∝ batch_size |

**實戰驗證**（你該做）：
```python
# max_seq_length 影響內存計算
Memory = batch_size × seq_length × hidden_size × 32 / 8
# 128: 8 × 128 × 768 × 32 / 8 = 400 MB ✓
# 256: 8 × 256 × 768 × 32 / 8 = 800 MB (會很擁擠)
# 512: 8 × 512 × 768 × 32 / 8 = 1.6 GB (即使沒 loss，也會 OOM!)

# learning_rate 的「大小」判斷
# 2e-5 = 0.00002 (微調用)
# 1e-4 = 0.0001  (太大，會震盪)
# 5e-6 = 0.000005 (太小，進度慢)
```

---

#### 🔧 Level 2：實戰必備（知道如何調校）

**何時用？** 當模型練不好、顯存還是不夠、或過擬合時。

| 參數                              | 現在設定 | 什麼時候要改？                                               |
| ------------------------------- | ---- | ----------------------------------------------------- |
| **num_train_epochs**            | 3    | 如果驗證準確率還在上升 → 加到 5；早就停滯了 → 減到 2                       |
| **gradient_accumulation_steps** | 2    | 顯卡塞不下更大 batch？公式：$Effective\ Batch = 8 \times 2 = 16$ |
| **warmup_steps**                | 100  | 訓練剛開始時，從 0 逐漸升到設定的學習率，防止「大腦抽筋」                        |

**梯度累積的救星公式**：
```
如果你想跑 batch_size=32 但 OOM，改法：
  per_device_train_batch_size = 8
  gradient_accumulation_steps = 4
  # 實際效果：每次更新用 8×4=32 樣本的梯度，但顯存只用 8 個樣本的空間！

不用梯度累積：1 epoch=1000 次更新
用梯度累積：1 epoch=250 次更新（但每次梯度來自 4 倍的數據）
→ 收斂慢點，但能用 8GB GPU 訓練
```

#### ⚡ Level 1 的深度理解（額外參考）

**為什麼 BERT 微調 learning_rate 要這麼小？**
```
- 從零訓練（從頭學）：LR 可用 1e-3～1e-4（大步子，快速探索）
- BERT 微調（只調整最後一層）：LR 用 2e-5～5e-5（小步子）
  
原因：BERT 已經在 102 億自然語言數據上訓練過了。
現在只是「輕輕推」它識別「情感」這一小件事。
大步子會摧毀它已學到的知識（Catastrophic Forgetting）。
```

**為什麼 batch_size=8？**
```
顯存容量：3050 有 8GB VRAM
單個樣本內存：128×768×32/8 = 50MB（僅僅 forward pass）
加上梯度、優化器狀態：×4～5 倍 = 200～250MB/樣本
8 個樣本：8 × 250MB = 2GB
加上其他開銷：總共 ≈ 4-5GB
留下 2-3GB 給操作系統 → 舒服訓練
```

### 推論時的完整流程

```python
# 第 1 步：加載模型和分詞器
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import torch

MODEL_PATH = "./final_model"
tokenizer = AutoTokenizer.from_pretrained(MODEL_PATH)
model = AutoModelForSequenceClassification.from_pretrained(MODEL_PATH)

# 第 2 步：輸入文本
text = "This movie is absolutely amazing! Highly recommend!"

# 第 3 步：分詞
inputs = tokenizer(
    text,
    max_length=128,
    padding='max_length',
    truncation=True,
    return_tensors='pt'
)
# inputs = {
#   'input_ids': tensor([[101, 2054, 3123, ...]))
#   'attention_mask': tensor([[1, 1, 1, ...]])
# }

# 第 4 步：推論
model.eval()  # 切換到評估模式（dropout 不生效）
with torch.no_grad():  # 不記錄梯度（推論時不需要）
    outputs = model(**inputs)
    logits = outputs.logits  # shape: [1, 2]

# 第 5 步：轉換為概率
probs = torch.softmax(logits, dim=1)  # [1, 2] → 每個類別的概率
pred_label = torch.argmax(probs, dim=1).item()  # 0 或 1

# 第 6 步：輸出
label_map = {0: "Negative", 1: "Positive"}
print(f"預測: {label_map[pred_label]}")
print(f"信心: {probs[0][pred_label].item():.2%}")
```

**這就是 pytorch_model.bin 的全部用途！**
